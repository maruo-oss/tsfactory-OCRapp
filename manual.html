システム概要

PDF形式の発注書を自動的に読み取り、Google Spreadsheetに発注データを保存する革新的なシステムです。

💡
AI駆動の自動処理
Gemini 2.5 Proを活用して、発注書の内容を高精度で構造化データとして抽出。手動入力の時間を大幅に削減します。

システムフロー

Google Drive (PDF保管)
     ↓
Google Apps Script (OCR処理)
     ↓  Gemini API
Google Spreadsheet (データ保存)
     ↓
Web UI (閲覧・編集)

このシステムは、拠点別に発注書PDFを管理し、AI技術で自動的にデータ化。Webインターフェースから簡単に確認・編集できます。

セットアップ

システムを稼働させるために必要な設定手順を説明します。

必要なもの

  • Googleアカウント - Google Apps Scriptを実行するために必要
  • Gemini API キー - Google AI Studioで取得
  • Google Driveフォルダ × 2 - 入力用と処理済み用
  • Google Spreadsheet - データ保存用

Google Apps Script プロジェクトの作成

  1. Google Apps Script にアクセス
  2. 「新規プロジェクト」をクリック
  3. 以下のファイルを作成：
    • Code.gs - メインスクリプト
    • index.html - UI HTML
    • appsscript.json - プロジェクト設定
  4. 各ファイルにコードを貼り付け

スクリプトプロパティの設定

スクリプトエディタで「プロジェクトの設定」→「スクリプト プロパティ」から以下を設定：

プロパティ名 説明 例
GEMINI_API_KEY Gemini APIキー AIza...
ROOT_IN_FOLDER_ID 処理対象PDFを配置するフォルダID 1Abc...
ROOT_PROCESSED_FOLDER_ID 処理済みPDFの移動先フォルダID 1Def...
SPREADSHEET_ID データ保存先スプレッドシートID 1Ghi...
💡
フォルダIDの取得方法
Google DriveでフォルダのURLを開き、https://drive.google.com/drive/folders/ここがフォルダID の部分をコピーします。

フォルダ構造の作成

Google Driveに以下の構造でフォルダを作成します：

📁 ROOT_IN_FOLDER (処理対象)
├─ 📁 東京
│ ├─ 📄 発注書_001.pdf
│ └─ 📄 発注書_002.pdf
└─ 📁 大阪
└─ 📄 発注書_003.pdf

📁 ROOT_PROCESSED_FOLDER (処理済み)
├─ 📁 東京 (自動作成)
└─ 📁 大阪 (自動作成)
⚠️
重要
処理対象PDFは必ず拠点名フォルダの中に配置してください。ルート直下に置いても処理されません。

プロンプトシートの作成

スプレッドシートに「プロンプト」という名前のシートを作成し、A1セルに以下のプロンプトを記述：

以下のPDF発注書から以下の情報をJSON形式で抽出してください：
- order_number: 発注番号/発注No/注文番号（見つからない場合は空文字列）
- order_date: 発注日（YYYY-MM-DD形式）
- maker_name: メーカー名
- shop_name: 店舗名
- delivery_destination: 納品先（株式会社/有限会社など）
- items: 商品明細の配列
  - product_code: 品番
  - product_name: 商品名
  - quantity: 数量
  - unit_price: 単価

このプロンプトはカスタマイズ可能です。必要に応じて抽出項目を追加・変更してください。

Webアプリとしてデプロイ

  1. スクリプトエディタで「デプロイ」→「新しいデプロイ」をクリック
  2. 種類を「Webアプリ」に設定
  3. 実行ユーザー: 「自分」を選択
  4. アクセスできるユーザー: 組織内（または必要に応じて設定）
  5. 「デプロイ」をクリックしてURLを取得
  6. 取得したURLをブックマークに保存
✅
セットアップ完了！
デプロイURLにアクセスすると、AI-OCR発注管理システムが起動します。

使い方

システムの基本的な操作方法を解説します。

1. 発注書の自動処理

PDFファイルの配置

Google Driveの ROOT_IN_FOLDER 配下に拠点名フォルダ（例：東京、大阪）を作成し、各拠点フォルダにPDF発注書を配置します。

自動取込の実行

Webアプリのヘッダーにある「自動取込実行」ボタンをクリックします。システムがGoogle Driveをスキャンし、新しいPDFを自動処理します。

⏱️
処理時間について
PDFの枚数やページ数によって処理時間が変わります。通常、1枚のPDFで10〜30秒程度かかります。

処理結果の確認

処理が完了すると、左側の発注書一覧に新しいデータが表示されます。処理済みのPDFは自動的に ROOT_PROCESSED_FOLDER に移動されます。

2. Web UIの操作

画面構成

  • 左サイドバー - 発注書一覧（拠点バッジ、ファイル名、発注日、明細数を表示）
  • 中央ペイン - 発注詳細情報（拠点、店舗名、納品先、メーカー、発注日、発注No、ステータス、ファイル名、備考、商品明細）
  • 右ペイン - PDFプレビュー（原本PDFをリアルタイム表示）

発注書の閲覧

  1. 左サイドバーから発注書を選択
  2. 中央ペインで詳細情報を確認
  3. 右ペインでPDF原本と照合

データの編集

  1. 発注書を選択
  2. 「編集」ボタンをクリック
  3. 必要な項目を修正（発注日、発注No、メーカー、ステータス、ファイル名、備考、商品情報など）
  4. 「保存」ボタンで変更を確定
⚠️
注意
「拠点」フィールドは編集できません。拠点を変更する場合は、Google Drive上でPDFを移動してください。

アーカイブ機能（完了管理）

  1. 処理が完了した発注書にチェック
  2. 「選択した発注書を完了にする」ボタンをクリック
  3. 確認ダイアログで「OK」
  4. アプリ内では非表示になりますが、データはスプレッドシートに保持されます
  5. ステータスを変更すれば再度表示可能

Excel出力と自動アーカイブ

  1. 出力したい発注書にチェック
  2. 「選択した店舗をExcel出力」ボタンをクリック
  3. 確認ダイアログで「OK」
  4. Excelファイルがダウンロードされ、自動的に「完了」ステータスになります
  5. アプリ内では非表示になりますが、データはスプレッドシートに保持されます
💡
便利な機能
ヘッダーの「表示フィルタ」で拠点別に絞り込み可能。「完了」ステータスのデータは自動的に非表示になります。

フィルタリング

ヘッダーの「表示フィルタ」ドロップダウンから拠点を選択：

  • 全拠点 - すべての発注書を表示
  • 東京 - 東京拠点のみ
  • 大阪 - 大阪拠点のみ

※「完了」ステータスのデータは自動的に非表示されます

データ構造

スプレッドシートに保存されるデータの構造を説明します。

V2スキーマ（17列構成）

スプレッドシートの「OrderData」シートには以下の列が保存されます：

列名 列 説明
branch_name A 拠点名（例：東京、大阪）
file_id B Google Drive ファイルID
file_name C ファイル名（編集可能）
status D 処理ステータス（未処理/処理中/確認中/完了/保留/要確認）
order_date E 発注日（YYYY-MM-DD形式）
maker_name F メーカー名
shop_name G 店舗名
product_code H 品番
product_name I 商品名
unit_price J 単価
quantity K 数量
line_total L 小計（単価×数量）
processed_at M 処理日時
delivery_destination N 納品先（株式会社/有限会社など）
order_number O 発注番号
comment P 備考（編集可能）
item_order Q 商品明細の順序（PDF順）
🔄
V1.0からの移行
V1.0（14列）からV2.0（17列）への移行は、Apps Scriptエディタで migrateToV2Schema() 関数を実行するだけで完了します。既存データは保持されます。

データの保存形式

1つの発注書に複数の商品がある場合、商品ごとに1行が作成されます。つまり、発注書の基本情報（拠点、ファイル名、発注日など）は各商品行で重複します。

branch_name | file_id | file_name     | ... | product_code | product_name
------------------------------------------------------------------------
東京        | abc123  | 発注書_001.pdf | ... | P001        | 商品A
東京        | abc123  | 発注書_001.pdf | ... | P002        | 商品B
東京        | abc123  | 発注書_001.pdf | ... | P003        | 商品C

この形式により、スプレッドシート上で柔軟にデータ分析やピボットテーブルが作成できます。

トラブルシューティング

よくある問題とその解決方法を紹介します。

PDFが処理されない場合

  1. スクリプトプロパティの確認
    • GEMINI_API_KEY が正しく設定されているか
    • ROOT_IN_FOLDER_ID が処理対象フォルダのIDか
    • SPREADSHEET_ID が正しいスプレッドシートのIDか
  2. Gemini API キーの有効性確認
    • Google AI Studioでキーが有効か確認
    • API利用制限に達していないか確認
  3. Drive API の有効化確認
    • appsscript.json で Drive API が有効になっているか確認
  4. フォルダ構造の確認
    • PDFが拠点フォルダの中に配置されているか（ルート直下ではない）
    • フォルダ名に特殊文字が含まれていないか

デバッグ方法

Apps Scriptエディタで以下の関数を実行して、システムの状態を診断できます：

debugFolderCheck(); // フォルダとファイルの診断

実行ログに詳細な情報が表示されます。「表示」→「ログ」で確認してください。

エラーが発生したPDF

処理中にエラーが発生したPDFは、ファイル名に 【ERROR】 プレフィックスが付きます。以下の原因が考えられます：

  • PDFが破損している
  • PDFのフォーマットが複雑すぎる
  • AIが内容を正しく解釈できなかった
  • API呼び出しが失敗した

エラーPDFは手動で内容を確認し、必要に応じてデータを手動入力してください。

データが表示されない

  1. ブラウザをリフレッシュ（F5）
  2. スプレッドシートに「OrderData」シートが存在するか確認
  3. スプレッドシートに実際にデータが保存されているか確認
  4. ブラウザのコンソール（F12）でエラーが出ていないか確認

編集が保存されない

  1. 「保存」ボタンをクリックしたか確認
  2. スプレッドシートへの書き込み権限があるか確認
  3. ブラウザのコンソールでエラーを確認
⚠️
権限エラーが出る場合
Webアプリのデプロイ設定で「実行ユーザー: 自分」になっているか確認してください。また、初回アクセス時に権限の承認が必要です。

パフォーマンスが遅い

  • 大量のPDFを一度に処理しない - 10〜20件ずつに分けて処理
  • 完了したデータはアーカイブする - 表示データ量を減らすことで高速化
  • ブラウザのキャッシュをクリア - 古いデータが残っている可能性

サポートが必要な場合

上記の方法で解決しない場合は、以下の情報を添えて開発者に連絡してください：

  • エラーメッセージ（スクリーンショット）
  • Apps Scriptのログ
  • 問題が発生した操作手順
  • 使用しているブラウザとバージョン
